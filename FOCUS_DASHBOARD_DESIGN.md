## 一、设计背景

### 1.1 模块定位

Focus Dashboard 是 IB-AMS 系统的「聚焦视图」，用于快速掌握**当前时刻 + 今天 + 近期**的工作状态。

**核心价值**:
- 一屏展示关键信息，减少上下文切换
- 强调「此刻该做什么」的即时性
- 独立展示各区域，无交叉干扰

### 1.2 数据层

- **数据来源**: 聚合 Calendar API + Tasks API + Module 4 触发器
- **同步策略**: 实时同步，聚焦视图数据不持久化
- **刷新机制**: 页面激活时 + 每 30 秒轮询

### 1.3 界面参考

- **UI 范本**: Notion Daily Notes + Google Keep
- **核心特性**:
  - 5 区域横向布局
  - 视内容自适应宽度
  - 最小信息密度原则
  - 无区域联动

## 二、核心设计决策

### 2.1 时间范围定义

| 区域 | 时间范围 | 说明 |
|------|----------|------|
| 今日日历 | 今天 00:00 - 23:59 | 完整一天 |
| 当下任务流 | 当前时刻 + 今天剩余 | 强调即时性 |
| 当下消息流 | 当前时刻之前未处理 | 待办提醒聚合 |
| 明日任务 | 明天 00:00 - 23:59 | 预览 |
| 本周任务 | 剩余工作日 | 含周一至周五 |

### 2.2 布局策略

```
┌─────────┬─────────┬─────────┬─────────┬─────────┐
│  今日   │  当下   │  当下   │  明日   │  本周   │
│  日历   │  任务流 │  消息流 │  任务   │  任务   │
│  视图   │  视图   │  视图   │  集合   │  集合   │
└─────────┴─────────┴─────────┴─────────┴─────────┘
      ↑          ↑          ↑          ↑          ↑
   自适应宽    自适应宽    自适应宽    自适应宽    自适应宽
```

**宽度规则**:
- 最小宽度: 200px
- 最大宽度: 400px
- 内容少时收缩，内容多时扩展
- 横向滚动时各区域独立

### 2.3 交互原则

| 原则 | 说明 |
|------|------|
| 独立展示 | 各区域无联动、无交叉筛选 |
| 点击跳转 | 点击区域内的任务/事件 → 跳转详情页 |
| 无拖拽 | 不支持跨区域拖拽 |
| 只读为主 | 聚焦视图以展示为主，少量快速操作 |

### 3.2 Task 模型（扩展状态）

```typescript
// Task 基础模型复用 DESIGN_NOTES_20260330.md
// 本模块关注状态流转

interface Task {
  id: UUID;
  title: string;
  plan_id: UUID;
  bucket: string;
  estimated_minutes: number;
  deadline?: Date;
  schedule_date: Date;
  
  // 状态相关
  status: 'pending' | 'in_progress' | 'completed';
  priority: 'P1' | 'P2' | 'P3' | 'P4';
  
  // 延后相关
  delayed_until?: Date;  // 延后目标日期

### 3.3 任务状态机

```
┌─────────┐    ┌─────────────┐    ┌───────────┐
│ pending │───▶│in_progress │───▶│ completed │
└─────────┘    └─────────────┘    └───────────┘
```

| 操作 | 状态转换 | 说明 |
|------|----------|------|
| 启动 | pending → in_progress | 开始处理任务 |
| 完成 | in_progress → completed | 任务完成 |
| 延后 | 更新 schedule_date | 留在今日列表，标记 delayed_until |
| overdue | 实时计算条件 | 不改变任务状态，只更新 schedule_date |

### 3.4 overdue 定义

overdue 是**条件**，不是独立状态：
```
isOverdue = deadline < now && status != completed
```

| 项目 | 说明 |
|------|------|
| 计算方式 | 实时计算，不存储字段 |
| 延后操作 | 只改 schedule_date，新值 ≤ deadline |
| deadline | 只能用户手动修改 |

## 五、调度逻辑

### 5.1 关键概念

| 概念 | 说明 | 可修改性 |
|------|------|----------|
| **schedule_date** | 计划执行日期 | 用户可改，Arrange 可调度调整 |
| **deadline** | 截止日期，用户手动设置 | **只能用户手动修改** |

### 5.2 约束规则

| 规则 | 说明 |
|------|------|
| Arrange 范围 | 只对 `deadline - schedule_date > 24hr` 的任务进行调度 |
| Arrange 调整范围 | 只能把 schedule_date 往前（更早）调整 |
| Deadline 不可改 | deadline 只能用户手动修改，系统调度不能改 |
| Overdue 约束 | 只修改 schedule_date，且新值 ≤ deadline |

### 5.3 调度分层

| 层级 | 名称 | 作用 |
|------|------|------|
| 第一层 | Arrange | 跨日任务安排，重新计算 schedule_date |
| 第二层 | Daily Settlement | 当日任务排序，不改变任务属性 |

### 5.4 Arrange

#### 目的
跨日任务安排，把任务分配到合适的日期

#### 触发时机
- 每日固定时间（如 6:00）
- 手动触发

#### 任务范围
- 当前周 + 下一周的所有 pending 任务
- 向后跨周（不超过下下周）
- **前提条件**：`deadline - schedule_date > 24hr`

#### 调度逻辑

```
输入：
- 当前周 + 下一周的 pending 任务
- 课表（各天的课程安排）
- 时间容器配置

前置条件检查：
- 只有 deadline - schedule_date > 24hr 的任务才参与调度

处理：
1. 遍历 pending 任务（满足前置条件）
2. 在 [today, deadline - 24hr] 窗口内查找匹配日期
3. 匹配规则：
   a. 如果当天课表有对应学科：
      - 时间容器未满 → 安排到学习时间容器
      - 时间容器已满：
        - P1/P2 → 安排到自由时间容器
        - P3/P4 → 安排到下一个学科日
   b. 如果当天没有对应学科（非学科日）：
      - P1/P2 → 安排到自由时间容器
      - P3/P4 → 安排到下一个学科日
4. 确定任务的 schedule_date

注意：只修改 schedule_date，不改变 deadline
```

### 5.5 Daily Settlement

#### 目的
调整当日任务列表顺序，判定当前任务

#### 触发时机
- 每次打开 App
- 每 30 分钟定时触发

#### 任务范围
- 当日 scheduled_date = today 的 pending 任务

#### 处理逻辑

```
输入：
- 当日 scheduled_date = today 的 pending 任务

处理：
1. 按优先级重新排序（P1 > P2 > P3 > P4）
2. 判定当前任务（队列第一个 pending 任务）
3. 不修改任何任务属性

输出：
- 排序后的任务列表
- 当前应该执行的任务
```

### 5.6 任务优先级排序

| 优先级 | 说明 |
|--------|------|
| P1 | 紧急任务，最优先 |
| P2 | 重要任务 |
| P3 | 一般任务 |
| P4 | 可延后任务 |

### 5.7 当下任务流的任务来源

当下任务流展示的任务 = Daily Settlement 排序后的结果

```
┌──────────────────────────────┐
│ ⚡ 当下任务                    │
│                              │
│ ● Math IA 选题 (P1)          │ ← 当前任务
│ ○ Chemistry 作业 (P2)         │ ← 队列中
│ ○ Physics 实验报告 (P2)       │
│ ○ EE 初稿修改 (P3)           │
└──────────────────────────────┘
```

```

### 6.8 延后处理

| 延后选项 | 行为 |
|----------|------|
| 1天 | schedule_date = 今天 + 1天 |
| 2天 | schedule_date = 今天 + 2天 |
| 1周 | schedule_date = 今天 + 7天 |

**延后标记**:
- 延后任务留在今日列表顶部
- 显示 `⏰ 已延后至 明天 (1)` 标记
- 括号内数字表示该日期下的延后任务数

### 6.9 当前任务判定

| 条件 | 判定结果 |
|------|----------|
| 在当前时间容器内，第一个 pending 状态任务 | → 当前任务（显示实心圆点 ●） |
| 当前无时间容器 | 显示「空闲时间」提示 |
| 当前任务已完成 | 自动切换下一个 pending 任务 |
| 无 pending 任务 | 显示「今日任务已全部完成」 |

### 6.10 交互

| 操作 | 行为 |

## 十二、数据来源与 API 集成

### 12.1 数据来源映射

| 区域 | 数据来源 | API |
|------|----------|-----|
| 今日日历 | Calendar API | `events.list()` Layer 0 + Layer 1 |
| 当下任务流 | Tasks API + 本地调度状态 | `tasks.list()` + `schedule_state` |
| 当下消息流 | Module 4 触发器 | `reminders.list()` + Gmail API |
| 明日任务 | Tasks API | `tasks.list()` filter by date |
| 本周任务 | Tasks API | `tasks.list()` filter by date range |

### 12.2 本地缓存 (SQLite)

```sql
-- Focus Dashboard 专用缓存
CREATE TABLE focus_cache (
  id TEXT PRIMARY KEY,
  region TEXT NOT NULL,
  data_json TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  expires_at TEXT NOT NULL
);

CREATE INDEX idx_focus_region ON focus_cache(region);
CREATE INDEX idx_focus_expires ON focus_cache(expires_at);
```

### 12.3 刷新策略

```
页面加载:
  1. 检查 focus_cache 是否存在且未过期
  2. 存在 → 直接渲染
  3. 不存在 → 并行请求所有区域数据 → 聚合 → 缓存 → 渲染

页面激活 (visibilitychange):
  1. 检查距上次刷新是否 > 10 秒
  2. 是 → 刷新当前区域数据
  3. 否 → 跳过

定时轮询 (30秒):
  1. 仅刷新「当下」相关区域（任务流、消息流）
  2. 明日/本周区域按需刷新
```

## 十五、与现有模块的关系

### 15.1 Module 1（日历模块）

- Focus Dashboard 依赖 Module 1 的日历数据
- 今日日历视图直接复用 Module 1 的数据展示逻辑

### 15.2 Module 2（任务模块）

- 当下任务流来自 Module 2 的调度结果
- 明日/本周任务视图是 Module 2 的聚合视图精简版
- 点击跳转至 Module 2 详情页
- 任务操作（启动/完成/延后）需要同步更新到 Module 2

### 15.3 Module 4（触发器模块）

- 当下消息流依赖 Module 4 的 Reminder 和消息解析
- Module 4 产生的数据实时推送到 Focus Dashboard
