  **第 一 部分  需求描述与规划**

## **1.1  核心问题：我想解决什么？**

IB 课程体系（IBDP）横跨六门学科加核心课程，是世界上认知负荷最高的高中体系之一。作为 IB 学生，你面临的核心管理挑战可以归结为以下四个根本矛盾：

| 问题维度 | 具体表现 | 根本原因 |
| :---- | :---- | :---- |
| 认知负荷过载 | IA × 6、EE × 1、TOK \+ CAS，任务量极大、碎片化严重 | 缺乏统一的任务可见性与优先级排序机制 |
| 管理与执行脱节 | 每天大量时间用于思考该做什么，真正产出时间严重压缩 | 计划工具与执行工具分离，切换成本高 |
| 防御性不足 | 突发临时作业、社团通知等轻易摘毁已有日程安排 | 日程为静态结构，缺乏动态愈合与自我修复能力 |
| 当下聚焦困难 | 多任务并发时注意力分散，无法进入深度工作状态 | 没有此刻唯一任务的清晰显示机制 |

## **1.2  达成目标：我希望实现什么？**

### **目标一：学术卓越**

* 确保所有 IA、EE 等长周期项目严格按里程碑交付，不遗漏任何单元考核复习节点

* EE（12个月）、IA × 6（跨学期）、TOK 论文与展览均纳入可追踪的阶段性任务系统

* 单元考核（每学期三次 × 六科）与学年模拟考前均有充分复习缓冲

### **目标二：管理自动化**

* 将时间分配的决策成本外包给算法，学生只负责定义规则和产出结果

* 避免手动日历排列，任务应自动找到最优时间槽

* 突发事件发生后，受影响任务自动愈合滑移，无需手动重排

### **目标三：心理稳态**

* 在任意时刻，学生只需看一眼时间线，就能知道当前唯一的最高优先级任务

* 防止学业焦虑和拖延症，通过系统性支持维护心理健康

* 4点至6点总结块、6:30导航时刻、9:30-11点结算时段必须被系统保护

  **第 二 部分  逻辑设计**

整个管理逻辑围绕一条连续的时间线展开，通过四个层级的叠加，形成防御性 \+ 动态愈合的完整管理架构。每一层都有明确的职责边界，层级之间形成稳定的优先级秩序。

## **2.1  四层时间线架构**

| 层级 | 名称 | 定义 | 优先级秩序 |
| :---- | :---- | :---- | :---- |
| Layer 0 | 静态层（确定性日程） | 不可移动的硬约束：上课、既定会议 | 最高，任何算法不得侵占 |
| Layer 1 | 防御层（框架性日程） | 约定时间段的行为类别：4-6点总结、6:30导航、9:30结算 | 次高，具有软防御性，极端情况下可弹性 |
| Layer 2 | 填充层（任务管理） | 产出导向的灵活任务块：IA撰写、EE章节、复习块 | 由 Priority × Urgency 决定，算法动态填入空隙 |
| Layer 3 | 触发层（提醒事项） | 特定时刻触发的消息/提醒，不占用时间块 | 辅助性，用于状态切换和及时响应 |

## **2.2  防御层：三大核心时间镑点**

框架性日程不指向单一任务，而是约定在此时段只做某类工作。以下三个时间镑点是本体系的核心支柱，必须被系统强力保护：

| 时间镑点 | 时段 | 行为类别 | 防御等级 | 备注 |
| :---- | :---- | :---- | :---- | :---- |
| 上课总结块 | 16:00 \- 18:00 | 消化今日 6 科知识点，整理笔记，完成当日作业 | Always Busy（硬防御） | 防止被其他任务占用 |
| 晚间导航时刻 | 18:30 \- 18:50 | 更新任务优先级，确定晚上要完成的最高价值任务 | Always Busy（硬防御） | 当下视角的每日入口 |
| 今日结算总结块 | 21:30 \- 23:00 | 产出归档：笔记/作业/试卷整理，标记任务完成状态 | 软防御，默认 Free | 触发愈合算法 |

## **2.3  填充层：任务属性模型**

| 属性 | 说明 | 示例 |
| :---- | :---- | :---- |
| Title（标题） | 明确的产出目标，非行为描述 | 完成 Chemistry IA 数据分析章节草稿 |
| Duration（预计耗时） | 算法据此寻找合适时间槽 | 90 分钟 |
| Deadline（截止日期） | 硬截止（Hard）或软截止（Soft） | 2025-10-15 (Hard) |
| Start Time（最早开始） | 可选，表示不早于此时开始 | 实验完成后（依赖关系） |
| Priority（优先级） | P1-P4，决定与其他任务的排序 | P1: 关键 / P2: 高 / P3: 中 / P4: 低 |
| Urgency（紧迫性） | 由 Slack \= Deadline \- Now \- Duration 动态计算 | Slack ≤ 24h 触发防御翻转 |
| Status（状态） | 待办 / 进行中 / 完成 / 逾期愈合 | 完成后触发归档 |
| Category（分类） | 区分学科来源和任务类型 | Chemistry / EE / CAS / TOK |

## **2.4  优先级 P1-P4 定义**

| 级别 | 标签 | 定义 | 算法行为 |
| :---- | :---- | :---- | :---- |
| P1 | 🔴 Critical | 关键：EE 提交前一周、IA 最终截止 | 自动覆盖低级别习惯与框架块（极端情况） |
| P2 | 🟠 High | 高：IA 各阶段截止、单元考复习 | 优先寻找最近空档插入，可覆盖 P3/P4 |
| P3 | 🟡 Medium | 中：日常作业、笔记整理 | 在 P1/P2 之外的空隙中填充 |
| P4 | 🟢 Low | 低：课外阅读、非紧迫 CAS 记录 | 仅在充裕空隙中安排，可被压缩 |

## **2.5  动态愈合（Healing）逻辑**

| 🔄 愈合算法核心逻辑 1\. 触发条件：结算块（21:30）运行扫描，检测今日已过去的任务块是否被手动标记为完成 2\. 若未完成：计算剩余耗时 \= 原 Duration \- 已完成量（若有记录），或全量保留 3\. 评估风险：检查 Slack \= Deadline \- Tomorrow \- Duration，若 Slack ≤ 0 触发 P1 升级警报 4\. 滑移安排：在未来时间线上寻找第一个满足条件的空隙，插入愈合任务块 5\. 通知用户：推送摘要今日 X 项任务已愈合，明日新增 Y 项任务，请确认 |
| :---- |

## **2.6  当下视角实现逻辑**

当下视角是本体系最核心的用户体验目标：在任意时刻，学生应能毫不稻豪地知道现在该做什么。

### **三步实现机制**

1. 18:30 导航时刻：将当晚最高价值任务拖入 Up Next 队列。系统无视其他所有优先级排序，在当前时刻后立即安排该任务

2. 进入专注块：系统自动全屏显示当前任务（名称 \+ 倒计时 \+ 番茄钟），同步通讯状态为深度工作中 / 请勿打扰，切断碎片化干扰

3. 紧急覆盖机制：若出现突发高优先级任务，使用 Up Next 紧急插入将其推至即时位置，不需要重新梳理整个任务列表

## **3.4  当下视角的 Reclaim 实现**

### **每日操作流程（18:30 导航时刻）**

4. 打开 Reclaim 的 Up Next 队列（侧边栏或 Slack 快捷命令 /reclaim next）

  **第 四 部分  自研工具架构（Google 生态）**

若 Reclaim.ai 的算法在 IB 多学科细颗粒度场景下无法满足需求，可参考以下自研方案。技术栈完全基于 Google 生态，运行成本极低。

## **4.1  系统总体架构**

| 层级 | 技术组件 | 职责 |
| :---- | :---- | :---- |
| 数据层 | Google Calendar API | 作为单一事实来源，存储所有日程与任务时间块 |
| 数据层 | Google Tasks API | 存储任务元数据（标题/截止日期/状态/优先级） |
| 数据层 | Google Sheets | 任务数据库（可视化编辑 \+ 历史记录） |
| 解析层 | Google Apps Script \+ LLM API | 解析 Gmail 邮件，提取任务信息，转化为结构化 JSON |
| 调度引擎 | Python CSP 优化器 / Apps Script | 核心调度算法，动态填充任务到 Calendar |
| 显示层 | Web App (Google Apps Script) | Focus Dashboard，全屏显示当前任务 \+ 番茄钟 |

## **4.2  核心调度算法设计**

| 算法目标与约束条件 目标函数：最小化最大逾期量 (Minimizing Maximum Lateness)   *Minimize: max(Lateness\_i)  where  Lateness\_i \= max(0, Finish\_i \- Deadline\_i)* 约束条件：   *1\. Start\_Time(Task\_i) \>= Current\_Time  （不安排过去）*   *2\. End\_Time(Task\_i) \<= Deadline\_i  （尽量在截止前完成）*   *3\. 任务块不重叠（Disjunctive Constraint）*   *4\. 避开 Layer 0（上课）和 Layer 1（三大镑点）*   *5\. 尊重任务依赖关系：IA 数据分析不早于实验完成* |
| :---- |

| 动态防御翻转（Defensive Flip）逻辑 For each task i:   *Slack\_i \= (Deadline\_i \- Current\_Time) \- Duration\_i*   *IF Slack\_i \<= 24h:*     *\-\> 将 Google Calendar 上该任务块状态：Free (透明) \-\> Busy (锁定)*     *\-\> 提升优先级至 P1，推送警报通知*   *IF Slack\_i \<= 0:*     *\-\> 触发逾期警报，推送愈合建议（列出可压缩的低优先级任务）* |
| :---- |

## **4.3  关键功能模块详细设计**

### **模块一：Gmail 邮件监听器（LLM 解析）**

* 使用 Google Pub/Sub 监听 Gmail 收件符1 (Google Apps Script watch() 方法)

* 当收到来自老师域名的邮件时，触发 LLM 解析（调用 Gemini API 或 DeepSeek API）

| LLM 解析 Prompt 模板 你是一个 IB 学生的学术助理。请从以下邮件中提取任务信息。 输出严格的 JSON 格式，不要任何额外文字： {   *task\_title: 简洁的产出目标描述,*   *deadline: YYYY-MM-DD,*   *estimated\_duration\_minutes: 数字,*   *subject: 学科名称,*   *priority: P1/P2/P3/P4,*   *notes: 重要细节* } 若无法确定截止日期，设为 null。 |

### **模块二：调度引擎伪代码**

## **4.5  自研 vs Reclaim.ai 决策树**

| 场景 | 建议方案 | 理由 |
| :---- | :---- | :---- |
| 刚进入 DP1 / 初次尝试数字化管理 | 先用 Reclaim.ai Lite (免费) | 零成本建立管理意识，验证工作流 |
| DP1 下学期，多 IA 并发管理 | 升级 Reclaim.ai Starter (学生折扣) | 自动调度价值显著，$5/月远低于自研成本 |
| 发现 Reclaim 任务依赖关系不足 | 扩展 Google Tasks \+ Apps Script 脚本 | 最小改动，补全依赖逻辑 |
| 需要 Gmail 自动解析任务 | 添加 Apps Script 邮件监听器 | 半自研，不动核心调度逻辑 |
| Reclaim 核心调度无法满足需求 | 实施完整自研方案（本第四部分） | 完全控制算法，但维护成本高 |
