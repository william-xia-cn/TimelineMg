## 一、设计背景

### 1.1 模块定位

Focus Dashboard 是 IB-AMS 系统的「聚焦视图」，用于快速掌握**当前时刻 + 今天 + 近期**的工作状态。

**核心价值**:
- 一屏展示关键信息，减少上下文切换
- 强调「此刻该做什么」的即时性
- 独立展示各区域，无交叉干扰

### 1.2 数据层

- **数据来源**: 聚合 Calendar API + Tasks API + Module 4 触发器
- **同步策略**: 实时同步，聚焦视图数据不持久化
- **刷新机制**: 页面激活时 + 每 30 秒轮询

### 1.3 界面参考

- **UI 范本**: Notion Daily Notes + Google Keep
- **核心特性**:
  - 5 区域横向布局
  - 视内容自适应宽度
  - 最小信息密度原则
  - 无区域联动

## 二、核心设计决策

### 2.1 时间范围定义

| 区域 | 时间范围 | 说明 |
|------|----------|------|
| 当下任务 | 当前时刻 + 今天剩余 | 强调即时性 |
| 今日 | 今天 00:00 - 23:59 | 完整一天 |
| 明日 | 明天 00:00 - 23:59 | 预览 |
| 本周 | 剩余工作日 | 含周一至周五 |
| 当下消息流 | 当前时刻之前未处理 | 提醒事项聚合 |

### 2.2 布局策略

```
┌────┬─────────┬─────────┬─────────┬─────────┬─────────┐
│    │  当下   │         │         │         │ 当下    │
│导  │  任务   │  今日   │  明日   │  本周   │ 消息流  │
│航  │         │         │         │         │         │
│栏  │  任务   │  任务   │  任务   │  周视图  │ 提醒    │
│    │  列表   │  列表   │  集合   │         │ 事项    │
│    │  (可    │         │         │         │         │
│    │  展开)  │         │         │         │         │
└────┴─────────┴─────────┴─────────┴─────────┴─────────┘
 ↑        ↑         ↑         ↑         ↑         ↑
固定宽   自适应宽  自适应宽  自适应宽  自适应宽  自适应宽
```

**宽度规则**:
- 最小宽度: 200px
- 最大宽度: 400px
- 内容少时收缩，内容多时扩展
- 横向滚动时各区域独立

### 2.3 交互原则

| 原则 | 说明 |
|------|------|
| 独立展示 | 各区域无联动、无交叉筛选 |
| 点击跳转 | 点击区域内的任务/事件 → 跳转详情页 |
| 无拖拽 | 不支持跨区域拖拽 |
| 只读为主 | 聚焦视图以展示为主，少量快速操作 |

## 三、数据模型

### 3.1 FocusDashboard 数据聚合

```typescript
interface FocusDashboardData {
  // 区域 1: 当下任务
  currentTaskFlow: {
    currentTask: Task | null;
    upcomingTasks: Task[];
  };

  // 区域 2: 今日
  todayTasks: {
    scheduledTasks: Task[];
    estimatedWorkload: number;
  };

  // 区域 3: 明日
  tomorrowTasks: {
    scheduledTasks: Task[];
    estimatedWorkload: number;
  };

  // 区域 4: 本周
  weekTasks: {
    tasksByDay: {
      [date: string]: Task[];
    };
    summary: WeekSummary;
  };

  // 区域 5: 当下消息流
  currentMessages: {
    reminders: Reminder[];
    pendingEmails: PendingEmail[];
  };
}

interface WeekSummary {
  totalCount: number;
  p1Count: number;
  p2Count: number;
  p3Count: number;
  p4Count: number;
  completedCount: number;
  overdueCount: number;
}
```

### 3.2 Task 模型

```typescript
interface Task {
  id: UUID;
  title: string;
  plan_id: UUID;
  bucket: string;
  estimated_minutes: number;
  deadline?: Date;
  schedule_date: Date;
  
  // 状态相关
  status: 'pending' | 'in_progress' | 'completed';
  priority: 'P1' | 'P2' | 'P3' | 'P4';
  
  // 延后相关
  delayed_until?: Date;
}
```

### 3.3 任务状态机

```
┌─────────┐    ┌─────────────┐    ┌───────────┐
│ pending │───▶│in_progress │───▶│ completed │
└─────────┘    └─────────────┘    └───────────┘
```

| 操作 | 状态转换 | 说明 |
|------|----------|------|
| 启动 | pending → in_progress | 开始处理任务 |
| 完成 | in_progress → completed | 任务完成 |
| 延后 | 更新 schedule_date | 留在今日列表，标记 delayed_until |
| overdue | 实时计算条件 | 不改变任务状态，只更新 schedule_date |

### 3.4 overdue 定义

overdue 是**条件**，不是独立状态：
```
isOverdue = deadline < now && status != completed
```

| 项目 | 说明 |
|------|------|
| 计算方式 | 实时计算，不存储字段 |
| 延后操作 | 只改 schedule_date，新值 ≤ deadline |
| deadline | 只能用户手动修改 |

## 五、调度逻辑

### 5.1 关键概念

| 概念 | 说明 | 可修改性 |
|------|------|----------|
| **schedule_date** | 计划执行日期 | 用户可改，Arrange 可调度调整 |
| **deadline** | 截止日期，用户手动设置 | **只能用户手动修改** |

### 5.2 约束规则

| 规则 | 说明 |
|------|------|
| Arrange 范围 | 只对 `deadline - schedule_date > 24hr` 的任务进行调度 |
| Arrange 调整范围 | 只能把 schedule_date 往前（更早）调整 |
| Deadline 不可改 | deadline 只能用户手动修改，系统调度不能改 |
| Overdue 约束 | 只修改 schedule_date，且新值 ≤ deadline |

### 5.3 调度分层

| 层级 | 名称 | 作用 |
|------|------|------|
| 第一层 | Arrange | 跨日任务安排，重新计算 schedule_date |
| 第二层 | Daily Settlement | 当日任务排序，不改变任务属性 |

### 5.4 Arrange

#### 目的
跨日任务安排，把任务分配到合适的日期

#### 触发时机
- 每日固定时间（如 6:00）
- 手动触发

#### 任务范围
- 当前周 + 下一周的所有 pending 任务
- 向后跨周（不超过下下周）
- **前提条件**：`deadline - schedule_date > 24hr`

#### 调度逻辑

```
输入：
- 当前周 + 下一周的 pending 任务
- 课表（各天的课程安排）
- 时间容器配置

前置条件检查：
- 只有 deadline - schedule_date > 24hr 的任务才参与调度

处理：
1. 遍历 pending 任务（满足前置条件）
2. 在 [today, deadline - 24hr] 窗口内查找匹配日期
3. 匹配规则：
   a. 如果当天课表有对应学科：
      - 时间容器未满 → 安排到学习时间容器
      - 时间容器已满：
        - P1/P2 → 安排到自由时间容器
        - P3/P4 → 安排到下一个学科日
   b. 如果当天没有对应学科（非学科日）：
      - P1/P2 → 安排到自由时间容器
      - P3/P4 → 安排到下一个学科日
4. 确定任务的 schedule_date

注意：只修改 schedule_date，不改变 deadline
```

### 5.5 Daily Settlement

#### 目的
调整当日任务列表顺序，判定当前任务

#### 触发时机
- 每次打开 App
- 每 30 分钟定时触发

#### 任务范围
- 当日 scheduled_date = today 的 pending 任务

#### 处理逻辑

```
输入：
- 当日 scheduled_date = today 的 pending 任务

处理：
1. 按优先级重新排序（P1 > P2 > P3 > P4）
2. 判定当前任务（队列第一个 pending 任务）
3. 不修改任何任务属性

输出：
- 排序后的任务列表
- 当前应该执行的任务
```

### 5.6 任务优先级排序

| 优先级 | 说明 |
|--------|------|
| P1 | 紧急任务，最优先 |
| P2 | 重要任务 |
| P3 | 一般任务 |
| P4 | 可延后任务 |

## 十二、数据来源与 API 集成

### 12.1 数据来源映射

| 区域 | 数据来源 | API |
|------|----------|-----|
| 当下任务 | Tasks API + Daily Settlement | `tasks.list()` |
| 今日 | Tasks API | `tasks.list()` filter by today |
| 明日 | Tasks API | `tasks.list()` filter by tomorrow |
| 本周 | Tasks API | `tasks.list()` filter by week |
| 当下消息流 | Module 4 触发器 | `reminders.list()` + Gmail API |

### 12.2 本地缓存 (SQLite)

```sql
CREATE TABLE focus_cache (
  id TEXT PRIMARY KEY,
  region TEXT NOT NULL,
  data_json TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  expires_at TEXT NOT NULL
);

CREATE INDEX idx_focus_region ON focus_cache(region);
CREATE INDEX idx_focus_expires ON focus_cache(expires_at);
```

### 12.3 刷新策略

```
页面加载:
  1. 检查 focus_cache 是否存在且未过期
  2. 存在 → 直接渲染
  3. 不存在 → 并行请求所有区域数据 → 聚合 → 缓存 → 渲染

页面激活 (visibilitychange):
  1. 检查距上次刷新是否 > 10 秒
  2. 是 → 刷新当前区域数据
  3. 否 → 跳过

定时轮询 (30秒):
  1. 仅刷新「当下」相关区域
  2. 其他区域按需刷新
```
